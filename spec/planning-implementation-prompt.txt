# Project Goal

Build a local prototype of an incremental data-sharing service.

The service reads changed `customers` and `cases` records from Postgres, writes those changes to local lake-style JSONL outputs, produces consumer-facing incremental change-feed artifacts, writes durable event files, and maintains a safe local checkpoint so committed changes are not skipped or duplicated because of timestamp ties or partial failures.

The prototype should demonstrate correctness and judgment around:

- incremental ingestion
- composite checkpointing
- deterministic ordering
- deterministic run IDs
- replay-safe outputs
- dry-run behavior
- practical crash-consistent local storage
- clear documentation
- testability

This is a local prototype. Do not overbuild with AWS services, Kafka, Airflow, Debezium, or cloud infrastructure. Those belong only in optional documentation.

# Checkpoint Contract

Store ingestion state locally at:

./state/checkpoint.json

The checkpoint must be per-table and must use a composite watermark, not timestamp alone.

Required shape:

{
  "customers": {
    "updated_at": "2026-03-31T12:00:00Z",
    "last_pk": 30
  },
  "cases": {
    "updated_at": "2026-03-31T12:00:00Z",
    "last_pk": 200
  }
}

For `customers`, `last_pk` means `customer_id`.

For `cases`, `last_pk` means `case_id`.

If no checkpoint exists, treat that as an initial empty watermark.

The checkpoint is a producer-side ingestion checkpoint. It is not a per-consumer cursor.

---

# Required Incremental Query Semantics

For each table, ingest rows where:

updated_at > checkpoint.updated_at
OR (
  updated_at = checkpoint.updated_at
  AND primary_key > checkpoint.last_pk
)

Rows must be processed in deterministic order:

ORDER BY updated_at ASC, primary_key ASC

This composite watermark is required to avoid missing rows that share the same timestamp.

For the initial run, when no checkpoint exists, ingest all rows from both tables in deterministic order.

# Practical Crash-Consistency Strategy

Implement staged writes.

The system must not leave itself in this dangerous state:

checkpoint advanced
but corresponding durable outputs missing or partially committed

The crash-consistency boundary includes:

- touched lake files
- durable event file
- share artifact file write/replace
- checkpoint file

The crash-consistency boundary does not include:

- stdout logging
- the HTTP response itself

Use a practical local strategy:

1. Compute deltas.
2. Compute manifest.
3. Compute `checkpoint_after`.
4. Compute deterministic `run_id`.
5. Write all new output content to a temporary run directory, for example `./tmp/runs/<run_id>/`.
6. Validate staged files.
7. Promote staged files into final paths where possible.
8. Write/replace the checkpoint file last using a temp-file-and-atomic-rename strategy.

Checkpoint writing should avoid partial JSON files:

- write `checkpoint.json.tmp`
- flush/sync if practical
- atomically rename to `checkpoint.json`

If anything fails before checkpoint update, leave the checkpoint unchanged and clean up temporary files where possible.

The system is not required to implement true multi-file atomic transactions across the filesystem.

However, it must prevent checkpoint advancement ahead of the durable outputs.

# Checkpoint Update Failure Recovery

Because the checkpoint is written last, there is a possible safe failure mode:

lake/share/event outputs written
checkpoint update fails

This should not cause data loss, but it may cause the next run to select the same delta again.

Use the deterministic `run_id` to reduce duplicate writes:

- If the same checkpoint and same source rows produce the same `run_id`
- and `./events/<run_id>.jsonl` already exists
- and the referenced output paths exist
- then treat the prior outputs as already materialized
- advance the checkpoint to the same `checkpoint_after` without rewriting duplicate lake/share outputs

Document this behavior if implemented.

If this recovery logic is not fully implemented, document the limitation clearly and ensure at minimum that the checkpoint never advances before durable outputs are written.

---

# Deterministic run_id

`run_id` must be deterministic from inputs.

The same source DB state and same checkpoint must produce the same `run_id`.

Implement this strategy:

1. Build a canonical JSON object containing:
   - `checkpoint_before`
   - for each table, the ordered delta row identities selected by the incremental query

2. Each row identity should include:
   - table name
   - primary key
   - updated_at

3. Serialize the canonical object with stable key ordering.

4. Hash it using SHA-256.

5. Use the hex digest, or a stable prefix of it, as `run_id`.

Because row identities are already ordered by:

updated_at ASC
primary_key ASC

the resulting `run_id` must be deterministic for the same source state and checkpoint.

Do not use UUIDs, random values, or current timestamps for `run_id`.

# Local Lake Output

Write valid JSONL files to the local filesystem.

Required layout:

./lake/customers/date=YYYY-MM-DD/data.jsonl
./lake/cases/date=YYYY-MM-DD/data.jsonl

Partitioning rule:

- `YYYY-MM-DD` is the UTC calendar date derived from each row’s `updated_at`.

If a single ingest run touches multiple dates, write multiple partitions.

The lake is an append-only CDC-style change log, not a current-state snapshot.

Each successful non-dry ingest appends the changed rows for that run to the appropriate date partition or partitions.

Records written to the lake for a run must correspond exactly to the incremental delta selected by the checkpoint query for that table.

Rerunning ingest with no source DB changes must not create duplicates because no new delta should be selected.

Deterministic ordering within each batch must follow:

updated_at ASC
primary_key ASC

A simple acceptable strategy:

- determine incremental delta for each table
- group changed rows by UTC `updated_at` date
- append changed rows as JSONL records to the correct partition files
- ensure a no-op run writes nothing

Lake records may be the row payload itself.

Example lake record for `cases`:

{"case_id":210,"customer_id":32,"title":"Billing dispute","description":"Customer reported duplicate invoice","status":"open","updated_at":"2026-03-31T12:00:00Z"}

# Consumer-Facing Incremental Share Artifact

After a successful non-dry ingest, produce deterministic consumer-facing incremental share artifacts per table.

Required output paths:

./share/customers/changes.jsonl
./share/cases/changes.jsonl

These files represent the latest successful incremental batch that an external consumer would read after a successful ingest.

They are not full snapshots.

They are deterministic companion change feeds for downstream synchronization.

Each record must be valid JSON and include at least:

table
op
primary key field
updated_at
run_id
schema_fingerprint
checkpoint_after
record

Primary key field:

- `customer_id` for customers
- `case_id` for cases

`op` may be:

upsert

`record` contains the full changed row payload.

Example share record for `cases`:

{
  "table": "cases",
  "op": "upsert",
  "case_id": 210,
  "updated_at": "2026-03-31T12:00:00Z",
  "run_id": "abc123",
  "schema_fingerprint": "def456",
  "checkpoint_after": {
    "updated_at": "2026-03-31T12:00:00Z",
    "last_pk": 210
  },
  "record": {
    "case_id": 210,
    "customer_id": 32,
    "title": "Billing dispute",
    "description": "Customer reported duplicate invoice",
    "status": "open",
    "updated_at": "2026-03-31T12:00:00Z"
  }
}

Determinism rule:

- Given the same source data and same checkpoint, the share artifact contents and record order must be byte-for-byte identical across repeated successful runs.
- Because `run_id` is deterministic from inputs, including it in the artifact does not break this requirement.

Ordering rule:

updated_at ASC
primary_key ASC

Zero-delta rule:

If a successful non-dry run has `delta_row_count = 0` for a table:

- do not create a new share artifact for that table
- do not modify the existing share artifact for that table
- report `share_path = null` in the manifest and durable event

This prevents no-op runs from creating misleading or duplicate share batches.

Dry-run rule:

- if `dry_run=true`, do not write share artifacts
